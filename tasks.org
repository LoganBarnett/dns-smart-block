** Query rewrites in ~common/src/db/queries.rs~

Each of these targets the ~get_metrics_stats~ function and exercises a
fundamental PostgreSQL planner limitation.  Indexes alone cannot fix any of
the three because:

- ~COUNT(DISTINCT col)~ always sorts/hashes to dedupe regardless of index.
- PostgreSQL has no native loose-index-scan, so ~DISTINCT ON (domain)~ with no
  ~WHERE~ clause sorts the whole table.
- An ~ORDER BY~ on a non-indexable expression (~CASE WHEN ...~) forces a sort
  even when the data is already in index order.

*** DONE [#A] Rewrite COUNT(DISTINCT domain) as GROUP BY subquery (6 call sites)
CLOSED: [2026-05-26 Tue]

*File:* ~common/src/db/queries.rs~ (six query bodies in ~get_metrics_stats~)

Affected queries (all six variants of "currently-valid domain count"):
- ~current_classifications_by_type~ (line 111)
- ~current_positive_by_type~ (line 130)
- ~current_negative_by_type~ (line 149)
- ~current_total~ (line 168)
- ~current_positive_total~ (line 179)
- ~current_negative_total~ (line 190)

Each runs ~SELECT [classification_type,] COUNT(DISTINCT domain) FROM
domain_classifications WHERE valid_on <= $1 AND valid_until > $1 [AND
is_matching_site = ?]~ and spills to disk (~6 MB temp file per query for the
grouped variants).  Replacing the ~COUNT(DISTINCT)~ with an inner
~GROUP BY~ lets PostgreSQL use ~HashAggregate~ instead of a disk-spilling
sort.

*Test first:*
The existing ~test_events_by_action_*~ tests in ~blocklist-server/src/db.rs~
already cover ~get_metrics_stats~ correctness.  Extend them (or add a new
test) to seed a domain that appears under multiple classification types
within the same validity window, and assert that the domain is counted once
per (domain, classification_type) pair — the same semantics as the original
~COUNT(DISTINCT)~.

*Fix:*
Grouped variant:
#+begin_src sql
SELECT classification_type, COUNT(*) AS count FROM (
  SELECT classification_type, domain
  FROM domain_classifications
  WHERE valid_on <= $1 AND valid_until > $1
  GROUP BY classification_type, domain
) d
GROUP BY classification_type;
#+end_src

Total variant:
#+begin_src sql
SELECT COUNT(*) FROM (
  SELECT domain
  FROM domain_classifications
  WHERE valid_on <= $1 AND valid_until > $1
  GROUP BY domain
) d;
#+end_src

Measured payoffs:
- Grouped: 683 ms → 155 ms (~4.4×).
- Total:   682 ms →  79 ms (~8.6×).
- Positive/negative filter variants:   ~17 ms (parallel seq scan picked).

*** DONE [#A] Rewrite events_by_action with lateral join over domains
CLOSED: [2026-05-26 Tue]

*File:* ~common/src/db/queries.rs:207~

The current query is:
#+begin_src sql
SELECT action::text as action, COUNT(*) as count
FROM (
  SELECT DISTINCT ON (domain) action
  FROM domain_classification_events
  ORDER BY domain, created_at DESC
) latest
GROUP BY action;
#+end_src

This is the single worst offender at 2117 ms per scrape — it sorts all 484k
events on ~(domain, created_at DESC)~ and writes a 35 MB temp file to disk
*every 10 seconds*.

The recursive-CTE loose-index-scan rewrite was tested and is actually slower
(3090 ms — PostgreSQL executes the subplan twice and still sorts).  The
working form is a lateral join driven off the ~domains~ table, which exploits
the existing ~idx_events_domain_created (domain, created_at DESC)~ index for
each per-domain lookup.

Caveat: 59373 unique domains appear in ~domain_classification_events~ vs
57244 in ~domains~ (a 4% gap).  Source of the gap is identified in the
companion ~DomainRequeue~ task below.  Fix that first, then this rewrite has
identical semantics to the original.

*Test first:*
Extend ~test_events_by_action_*~ in ~blocklist-server/src/db.rs~ with a case
where ~domains~ has fewer rows than the event-domain set (e.g. a requeue
without prior domain upsert) — and assert the rewrite still produces the
same counts after the ~DomainRequeue~ upsert fix lands.

*Fix:*
#+begin_src sql
SELECT t.action::text AS action, COUNT(*) AS count
FROM domains d
CROSS JOIN LATERAL (
  SELECT action FROM domain_classification_events e
  WHERE e.domain = d.domain
  ORDER BY e.created_at DESC LIMIT 1
) t
GROUP BY t.action;
#+end_src

Measured payoff: 2117 ms → 631 ms (~3.4×).  Eliminates the disk sort entirely
(plan switches to ~HashAggregate~).

*** DONE [#A] Rewrite get_blocked_domains as UNION with NOT EXISTS
CLOSED: [2026-05-26 Tue]

*File:* ~common/src/db/queries.rs:17~

The current query uses ~DISTINCT ON (domain) ... ORDER BY domain,
CASE WHEN classification_type = $1 THEN 0 ELSE 1 END~ to express "per-type
record wins, fall back to 'all' override".  The ~CASE~ expression is not
indexable, so PostgreSQL always falls back to an external merge sort
(6 MB temp file).  Called by Blocky's blocklist refresh — much less often
than ~/metrics~, but still wasteful and gets worse as the table grows.

*Test first:*
The existing ~test_get_blocked_domains_specific_type_*_wins_over_all_*~
tests in ~blocklist-server/src/db.rs~ already cover the override semantics.
Verify the rewrite produces identical results on the existing test cases.

*Fix:*
#+begin_src sql
WITH per_type AS (
  SELECT domain, is_matching_site
  FROM domain_classifications
  WHERE classification_type = $1
    AND valid_on <= $2 AND valid_until > $2
),
all_override AS (
  SELECT a.domain, a.is_matching_site
  FROM domain_classifications a
  WHERE a.classification_type = 'all'
    AND a.valid_on <= $2 AND a.valid_until > $2
    AND NOT EXISTS (SELECT 1 FROM per_type p WHERE p.domain = a.domain)
)
SELECT domain FROM per_type      WHERE is_matching_site = true
UNION
SELECT domain FROM all_override  WHERE is_matching_site = true
ORDER BY domain;
#+end_src

Measured payoff: 496 ms → 34 ms (~14×).

** Memory leak

*** TODO [#A] log-processor leaks memory at ~150 MB/min until restart

*Service:* ~dns-smart-block-log-processor~ (binary in ~log-processor/~).

Surfaced while watching silicon's overall memory after the Performance
deploys went out.  Each deploy reset the log-processor to a few MB and
memory dropped from ~50% to ~30% host-wide; between deploys the service
climbs back up.  Measured on silicon shortly after a deploy:

#+begin_example
dns-smart-block-blocklist-server    1.4 MB   (Peak: 2 MB)     up 11 min
dns-smart-block-log-processor       1.6 GB   (Peak: 1.6 GB)   up 11 min
dns-smart-block-queue-processor    12.5 MB   (Peak: 15.8 MB)  up 11 min
#+end_example

Peak == current → it's still growing.  Extrapolating from 1.6 GB in 11
minutes, the service would consume the whole host within an hour or so
were systemd not eventually intervening (or were the host not running
other things).  This is the source of the host-wide memory drop the user
observed at each deploy this morning; it is not a one-time accounting
artifact.

The other two services in the same crate workspace (blocklist-server,
queue-processor) sharing most of the same dependency tree do *not* leak,
so the cause is almost certainly local to log-processor's hot loop
(~log-processor/src/main.rs~ and ~log-processor/src/log_source.rs~) or
to a dependency only it uses (e.g. ~notify~, ~async-stream~) — not in
the shared ~common~ crate or ~async-nats~ alone.

The hot loop is small: ~journalctl -f -u blocky~ as a subprocess streamed
line-by-line through ~tokio::io::BufReader~, each line passed to
~parser.parse_log_line~, matched lines published to NATS via
~queue.publish_domain~.  Suspected culprits, in order of likelihood:

1. *Subprocess output buffering.*  ~log_source.rs~ reads ~child.stdout~
   through ~BufReader::new(stdout).lines()~.  If the subprocess emits
   lines faster than the parser+publisher consumes them, ~BufReader~
   accumulates.  Need to confirm there is no unbounded internal queue.
2. *Regex captures retained.*  ~parser.parse_log_line~ may be cloning or
   capturing more than the matched domain string per line.  Need to
   audit whether anything per-line escapes the loop iteration.
3. *NATS publisher state.*  ~async-nats~ ~publish~ is fire-and-forget;
   if publishes are queued and the server lags, the client-side queue
   could grow without bound.  Check whether there is backpressure or a
   bounded send buffer.

*Test first:*
The leak is observable, not deterministic-input-driven, so a unit test
is hard.  Two viable approaches:

1. *Load-driven integration test* — spawn the log-processor against a
   mock journal source that emits a large fixed number of lines (e.g.
   1M synthetic blocky log lines), record RSS at start and end, assert
   the delta is below a sane bound (e.g. < 50 MB regardless of input
   size).  Heavy to set up but reproduces the live signal.
2. *Profiling-driven* — run the binary under ~heaptrack~ (or
   ~jemalloc~'s built-in profiling) against the same synthetic stream
   and inspect the allocation graph for the dominant retain path.  No
   test artifact, but it tells you precisely which line allocates the
   non-reclaimed memory.

Start with (2) — it's cheap and points directly at the culprit.  Add (1)
as a regression guard once the fix is in.

*Fix:*
Depends on which suspect (1)/(2)/(3) above turns out to be the cause.
Likely candidates: bound the ~BufReader~ internal buffer; switch from
~async-stream~ ~yield~ to direct channel send with bounded capacity;
periodically call ~regex::Regex::clear_cache~ or replace per-line
allocations with reused buffers; use ~async-nats~'s publish-with-ack
form (or a bounded ~mpsc~ in front of it) if the unbounded fire-and-
forget queue is the issue.
